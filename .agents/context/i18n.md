# i18n Context

The frontend supports full bilingual en/ar (English and Arabic) UI.

## Key Files

| File | Purpose |
|---|---|
| `FrontEnd/src/i18n/translations.ts` | Master translation dictionary — all UI strings, both languages |
| `FrontEnd/src/i18n/useI18n.ts` | Custom hook — returns `t(key)` translation function |
| `FrontEnd/src/i18n/i18nStore.ts` | Zustand store — holds `language` (en|ar) and `setLanguage` |
| `FrontEnd/src/i18n/I18nBootstrap.tsx` | App-level initialization, sets `dir` on `<html>` for RTL |
| `FrontEnd/src/i18n/types.ts` | TypeScript types for translation keys |

## Usage in Components

```tsx
import { useI18n } from '../i18n/useI18n';

const { t, language } = useI18n();
return <h1>{t('employees.title')}</h1>;
```

## RTL Support

Arabic requires right-to-left layout. `I18nBootstrap` sets `document.documentElement.dir = 'rtl'` when language is `ar`. Ant Design's `ConfigProvider` is configured with the appropriate locale.

## Rules for Adding UI Text

- **Never hardcode English strings** in JSX — always use `t('some.key')`.
- Add both `en` and `ar` values to `translations.ts` for every new key.
- Use dot-notation namespacing by domain: `leaves.request_leave`, `payroll.run_details`, `employees.status.active`.
- For bilingual data from the backend (e.g., `EmployeeProfile.name_en` / `name_ar`), select the correct field based on `language` from `useI18n`.
- Ant Design components that accept `locale` props should receive the locale from `i18nStore`.

## Backend Bilingual Fields

Some backend models store both English and Arabic values:
- `EmployeeProfile`: `name_en`, `name_ar`, `position_en`, `position_ar`
- `Department`, `Position`: `name_en`, `name_ar`

When displaying these in the frontend, read the correct field based on active language.
